====================================================
0538pay 后端技术选型（B 方案 · 独立重写 · Go）
====================================================
定位：B 方案 = 不复用彩虹易支付（epay）的 PHP，后端用独立技术栈从零重写，
      功能对齐 epay，但语言/框架/表结构/插件契约全部自研，最大化拉开与 epay 的
      代码相似度，规避抄袭风险。
最后更新：2026-07-17
后端语言：Go（完整后端全部用 Go 完成开发）。
决策依据：支付平台压力集中在高并发 I/O 回调 + 长期稳定运维，正是 Go 强项；
          Java 最大优势(官方 SDK 全)在易支付聚合场景里发挥有限；
          金额精度用 shopspring/decimal + 工程纪律解决，非语言硬伤。

----------------------------------------------------------------
〇、总原则
----------------------------------------------------------------
- 前端不动：admin-web 仍是 Vue 3 + TypeScript + Vite（已完成的原型不重写）。
- 后端独立重写：完整 Go 技术栈，与 epay(PHP) 语言层彻底分离。
- 金额一律 shopspring/decimal（全局禁止 float64/float32 参与金额计算）。
- 数据表 / API 契约 / 插件框架均自研，不 1:1 复制 epay 的表名、字段名、插件约定。

----------------------------------------------------------------
一、各部分用什么语言 / 技术栈（核心速查表）
----------------------------------------------------------------
【前端 · 官网 + 后台 + 控制台 + 商户中心】
  语言：TypeScript
  框架：Vue 3 + Vite + Pinia + Vue Router + Tailwind
  状态：已完成原型（全 mock），接后端时把 mock 层换成真实 HTTP 调用。

【后端 · 主服务（业务 + 支付核心）】
  语言：Go（推荐 Go 1.22+）
  框架/库：
        - HTTP/路由：Gin（REST JSON API）
        - ORM：GORM（MySQL 驱动）
        - 鉴权：JWT（golang-jwt/jwt），对齐 epay 后台/商户/控制台三端角色 + RBAC 中间件
        - 参数校验：go-playground/validator（Gin 内置集成）
        - 配置：Viper；日志：zap；错误封装：统一 Result/errcode
        - 事务：GORM Transaction（分账/退款/结算等强原子场景手动管控事务边界）
  金额：github.com/shopspring/decimal（金额类型全局统一，禁用 float）
  加密/签名：Go 标准库 crypto/*（md5/sha256/rsa）+ encoding（MD5/RSA/SHA 签名验签）

【数据库】
  主库：MySQL 8.x（表结构自研，不复制 epay 的 29 张表）
  缓存/幂等/限流/分布式锁：Redis（go-redis/redis 客户端）

【支付插件系统 · 51 渠道对接】
  语言：Go（与主服务同栈）
  设计：自研插件抽象——定义统一 PaymentChannel interface（下单/查单/退款/回调验签/关单），
        各渠道实现该接口 + init() 注册到全局 registry。不复刻 epay 的 xxx_plugin.php 约定。
  对接方式：易支付类接口本就是「拼参数 + MD5/RSA 签名 + HTTP POST」，用 net/http + 自研签名
        工具封装即可；主流渠道(支付宝/微信/银联)可用优质社区 SDK（如 smartwalle/alipay、
        wechatpay-go）。

【异步任务 / 定时对账】
  语言：Go
  技术：robfig/cron（或 asynq + Redis 做可靠队列），对齐 epay cron.php 的对账、
        超时关单、结算跑批；异步 notify 重试用队列 + 幂等键。

【构建 / 部署】
  依赖管理：Go Modules（go.mod）
  构建产物：单二进制（CGO_ENABLED=0 静态编译），拷贝即跑
  运行环境：无需 JVM/PHP，脱离 phpStudy；本地装 Go 工具链 + MySQL + Redis
  部署：二进制直跑 或 Docker 容器化（镜像极小）

----------------------------------------------------------------
二、为什么后端选 Go（专业理由）
----------------------------------------------------------------
1. 高并发回调：支付系统压力集中在海量第三方异步 notify + 商户查询，是 I/O 密集场景，
   goroutine 天生擅长，用最少代码和资源扛住，无需线程池调优。
2. 部署运维碾压：编译成单个静态二进制，拷贝即跑，无 JVM、无 GC 调优、无依赖地狱——
   对小团队/个人运营是实打实的省心红利。
3. 资源占用低：同样机器比 JVM 扛更多连接，服务器成本更省。
4. 长期可维护：语法简单、显式、无框架魔法（无 Spring 那种注解/反射黑箱），
   代码好读、三年后回来仍看得懂。
5. 规避抄袭：epay 是 PHP，Go 与之跨语言，查重基本零命中（与 Java 打平）。
6. 金额精度：shopspring/decimal 是 Go 生态事实标准，稳定可靠，配合“全局禁用 float”
   的工程纪律，与 Java BigDecimal 无实质差距。

（备选 Java：仅当需对接大量只提供 Java 官方 SDK 的冷门渠道、或团队已有 Java 班底、
  或进持牌金融机构强制 Java 时才回头选。以上均不符合本项目。）

----------------------------------------------------------------
三、抄袭风险红线（Go 重写时必须遵守）
----------------------------------------------------------------
- 插件框架架构自己设计（Go interface + registry 注册），不照抄 epay 的 xxx_plugin.php
  钩子命名与目录约定。
- 数据表名 / 字段名做一次改写，不 1:1 复制 epay 的 29 张表。
- 支付宝/微信等对接代码天然会和 epay 相似（同一网关的签名/参数是网关规定的，属通用表达，
  可辩护）；风险点不在这，而在“插件注册契约、目录结构、自研命名”是否照抄 epay 自研框架。
- 文档/文案/提示语不逐字照搬 epay。
- 保留 git 提交历史，作为独立开发过程的证据。

----------------------------------------------------------------
四、金额与安全纪律（Go 无语言级 BigDecimal，靠纪律补齐）
----------------------------------------------------------------
- 金额类型：一律 decimal.Decimal（DB 用 DECIMAL 列），严禁 float64 参与任何金额运算。
- 回调幂等：notify 用「订单号 + 渠道流水号」幂等键 + Redis 锁 / 唯一索引，防重复通知。
- 验签必做：所有回调先验签再处理；金额、订单状态二次校验，防篡改。
- 事务边界：下单/支付成功/分账/退款/结算显式包事务，保证原子性。

----------------------------------------------------------------
五、待定项（后续拍板）
----------------------------------------------------------------
- 金额存储：DECIMAL(如 DECIMAL(18,4)) 列 —— 精度位数待定。
- 定时/队列：robfig/cron（简单）vs asynq（可靠重试队列）—— 按 notify 重试可靠性要求定。
- 渠道分级：51 插件按“能否拿到商户资质 + 接口复杂度”分 P0/P1/P2，先做主流 3~5 个。
- 单体 vs 微服务：起步建议单体（Go 单工程按 domain 分包 internal/），跑通再按热点拆分
  （如高压回调网关未来可独立服务）。
- 项目结构：倾向 internal/ 分层（handler / service / repository / channel 插件），
  cmd/ 放入口，pkg/ 放可复用工具。
