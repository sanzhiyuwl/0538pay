【签名 / 安全】
  ✅  MD5 签名验签（V1：pid+key，ksort 去空去 sign）
      —— pkg/sign 逐字节对齐 epay getSignContent+makeSign+verifySign，
         基准由 epay php8.2 现场计算，Go 单测 3 个全绿(2026-07-21)。
  ✅  RSA SHA256 签名验签（V2）—— pkg/sign/rsa.go [真接口,2026-07-21]
        —— MakeRSA/VerifyRSA 复用 V1 的 Content() 待签名串 + SHA256WithRSA + base64,密钥单行base64/PKCS#8(对齐 epay pemToBase64)。
           GenerateRSAKeyPair 生成2048位。单测4绿(round-trip/篡改/错误密钥/待签名串与V1一致)。
           双向密钥流向对齐epay:下单商户私钥签→平台用商户公钥(pre_user.publickey)验;回调平台私钥签→商户用平台公钥验。
  ✅  keytype 分派 + timestamp 防重放（V2）—— Submit.verifySubmitSign [真接口]
        —— keytype=1 强制RSA(否则拒),keytype=0 兼容模式按 sign_type 选。RSA 校验 timestamp ±300s 防重放。
           真跑:keytype=1+MD5拒绝、RSA正确签名通过、无timestamp拒、过期timestamp拒、错误私钥拒、权限开通后RSA下单出收银台。
           商户端 RSA 密钥管理:POST /apikey/rsa 生成(私钥一次性返回不落库)+ PUT /apikey/keytype 切模式。MerchantApi.vue 切真接口。
  ✅  下单幂等（uid+out_trade_no）—— submit 幂等 + 参数变化拦截；回调侧条件 UPDATE 幂等改单
  ✅  回调金额/状态二次校验防篡改 —— 回调金额 decimal 精确比对订单金额，不一致拒绝
  —— MD5验签 + 下单/回调双向幂等 + 回调金额校验已落地(阶段A1+A4)；RSA/timestamp 待 V2。

================================================================
★ 结论与差距总结
================================================================
1. 精仿"界面层"≈90% 完成：admin 后台、商户中心、官网 UI 基本全对齐 epay 并超出
   (多了 SaaS 控制台 + RBAC + CMS)。这是我们最扎实的部分。

2. 精仿"能力层"≈5% 完成：epay 真正的价值(收单/渠道/回调/结算/代付/分账/定时)
   在我们这里几乎全是 mock。后端只有 登录+订单列表+商户列表 3 个只读真接口。

3. 最大且唯一的关键缺口 = 收单主链(第一/二/三部分)：
   下单→选通道→插件下单→收银台→回调验签→改单→入账→通知商户→结算/代付/分账→定时。
   这条链一通，0538pay 才从"支付管理界面"变成"支付系统"。

4. 补齐顺序(对应 开发进度总览.txt 的阶段 A~E)：
   A 收单核心MVP(mock渠道跑通全链路) → B 真实渠道3~5家 → C 后台业务域切真接口
   → D 商户中心接后端 → E 定时任务+收尾。
   横向(后台页面切真接口)价值不掉、可后置；纵向(收单链路)是未验证的最大风险,先啃。

5. 不追求 1:1 的项：epay 11 套首页模板(我们 1 套精品)、51 插件全做(按资质分级)、
   单管理员权限(我们升级为 RBAC)、投诉功能(epay 源码本身该文件缺失,核对后再定)。
