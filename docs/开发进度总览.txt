  ★★ 路线图（2026-07-21 结合 epay 功能重排，按优先级）★★
  ================================================================
  判断依据：epay 的核心价值是【收单主链】——
     商户调 API → 渠道下单 → 收银台/二维码 → 第三方回调 notify →
     订单更新 → 异步通知商户 → 入账/结算。
  我们四端界面已全有，但这条链一条没通，后端目前只有 登录+订单查询+商户查询
  三个只读接口。故以下以"打通收单主链"为主轴；后台其余页面切真接口是配套，非主线。

  ----------------------------------------------------------------
  阶段 A — 收单核心 MVP（最高优先，先啃风险）
  ----------------------------------------------------------------
  目标：一笔钱能真正从下单走到支付成功回调。先用【自建 mock 渠道】跑通全链路，
        不依赖真实第三方资质。做完这步，0538pay 才第一次是"支付系统"而非"支付管理界面"。
    ⬜ A1 下单接口 POST /api/pay/submit（易支付协议兼容：pid+参数+MD5 签名验签、
          幂等、金额 decimal）
    ⬜ A2 渠道插件契约落地：PaymentChannel interface（下单/查单/退款/回调验签/关单）
          + registry，先实现一个 mock 渠道（骨架里 internal/channel 已起了抽象，需补实现）
    ⬜ A3 收银台/二维码中间页：下单后返回支付链接或二维码
    ⬜ A4 回调 POST /api/pay/notify/{trade_no}：验签 → 幂等改单状态 → 触发商户异步通知
    ⬜ A5 商户异步通知 do_notify：GET/POST 商户 notify_url，判返回含 success，
          失败进重试队列
    ⬜ A6 订单入账 + 资金流水记录（pay_record 表）
    验收：mock 渠道下单 → 模拟回调 → 订单变已付 → 商户 notify 收到。

  ----------------------------------------------------------------
  阶段 B — 真实渠道接入（P0 主流 3~5 家）
  ----------------------------------------------------------------
    ⬜ B1 支付宝（当面付/网页，smartwalle/alipay）
    ⬜ B2 微信（Native/JSAPI，wechatpay-go；仓库已有 wechatpay 相关 skill 可复用）
    ⬜ B3 个人免签一家（vmq 类）作无资质兜底
    ⬜ B4 每家按 A2 interface 实现 + 后台通道配置页（config JSON 密钥表单）切真接口

  ----------------------------------------------------------------
  阶段 C — 后台业务域横向切真接口（配套收单，按已固化五层样板）
  ----------------------------------------------------------------
    ⬜ C1 通道管理 channels（B 阶段依赖它配密钥，优先）
    ⬜ C2 资金流水 record / 结算 settle + 批次 batch
    ⬜ C3 代付 transfer / 分账 profit-sharing
    ⬜ C4 风控 risk / 黑名单 blacklist / 域名 domain
    ⬜ C5 统计 statistics / 操作日志 log / 邀请码 regcode+invitecode
