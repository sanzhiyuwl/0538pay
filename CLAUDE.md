# 0538pay 项目规范

本项目目标：**1:1 精致还原彩虹易支付（epay）到 0538pay**。宁可慢工出细活，不允许攒问题、堆返工。

参考源码位置：`f:/phpstudy_pro/WWW/epay.test`（彩虹易支付 epay 原始 PHP 系统，作为功能与业务的权威对照基准）。

> **每次打开或新建 Claude Code 窗口，都必须先读并遵守本规范。** 本文件每个会话自动加载、优先级最高，下面的强制交付门槛对所有会话一律生效，不因换了窗口而重置。

## 开工仪式（每个新窗口/新会话动手前必须先走完，不可跳过）

> **触发词：用户发「开工」（或「新窗口」「开始」）时，立即执行本仪式，无需进一步确认。** 新窗口/新会话即使用户没喊，动手前也必须先走完本仪式。

在开始任何任务、写任何代码之前，**先按序完成下面三步，并向用户简短复述后再等派活**：

1. **读并遵守本规范 `CLAUDE.md` 全文**，重点吃透：1:1 精仿 epay 的目标、参考源码位置 `epay.test`、下面三道强制交付门槛（关卡一 epay 完整功能对比 / 关卡二 自检修复 / 关卡三 回写进度）。
2. **读记忆索引 `MEMORY.md` 及其中相关记忆文件**，尤其：
   - `dev-progress-master` → `docs/开发进度总览.txt`（判断「做到哪/下一步」先读它）
   - `epay-feature-parity-checklist` → `docs/epay功能对比清单.txt`（精仿完整度唯一权威）
   - `self-check-after-feature`（每写完功能必走的自检流程）
   - `admin-ui-design-system`、`css-tokens-are-oklch`（UI 与令牌规范）
   - `progress-docs-two-axis-rule`（回写两轴分工铁律）
3. **向用户复述关键约束 3~5 条**（确认已理解交付门槛与两轴回写），然后再接任务。

复述完成前不动手。任务进行中若跨越多步，按需回查上述记忆与文档。

## 强制交付门槛（每次写完代码必须走完，全绿才算完成）

一个功能/页面只要动了代码，交付前**必须**按序走完下面两道关卡。任一不通过 = 未完成，当场修复后重跑，循环到干净。不要等用户来指问题。

### 关卡一：与 epay 完整功能对比

写功能时和交付前，都要和 epay 做**完整**功能对比，而不是只看表面：

1. 找到 epay 中对应的源码（`epay.test/admin/*.php`、`epay.test/user/*.php`、插件、模板等），逐项比对。
2. 对齐清单：
   - **字段**：数据字段名、含义、类型是否一致，有无遗漏。
   - **状态字典**：订单/结算/审核等状态的取值与流转是否一致。
   - **操作项**：列表行操作、批量操作、表单动作是否齐全。
   - **业务规则**：金额计算、费率、风控、校验、边界处理是否与 epay 行为一致。
   - **交互流程**：完整走一遍用户路径，确认与 epay 语义一致，mock 数据业务自洽。
3. 发现缺失或偏差 → 补齐/修正后再进入关卡二。

### 关卡二：自检与修复

按序走完，全绿再交付（详见记忆 `self-check-after-feature`）：

1. **构建**：`cd admin-web && npm run build`（vue-tsc 严格 noUnusedLocals，未用 import/type 删干净）。
2. **对照设计规范**：逐条比 UI 设计规范（内容卡片禁 border/ring/shadow/圆角，用浅灰底 `bg-muted/40` 靠色差区分；表格默认左对齐；金额 `¥` 弱化 + `tabular-nums`；用到的颜色类确认已在 style.css 注册；列表 Panel 带主标题 title 不用 flush；开关用 `<Switch>` 组件）。
3. **对照 epay 真实业务**：即关卡一的结果落到代码，字段/状态/操作与 epay 一致。
4. **交互与边界**：分页 clamp、筛选后选中残留清空、菜单关闭、行操作菜单方向自适应、带图标输入框 `!pl-9`。
5. **真跑一遍**：`npm run dev` 起服务，`curl` 目标路由 + .vue/.ts 模块确认 HTTP 200 且编译产物无 error、dev log 无报错。检查完停掉临时 server。

发现问题 → 当场改 → 重新构建验证 → 循环到干净。

### 关卡三：回写进度文档（收尾，不可跳过）

两道关卡全绿后，**必须**按下面两轴分工回写留痕，才算真正完成。不管新窗口还是老窗口，只要完成了功能就必须回写。没回写 = 未完成。

两个进度文档各管一轴，不要混写（这是刻意分工，别把功能明细塞回总览）：

1. **功能完成度 → `docs/epay功能对比清单.txt`（每次做完功能都要更新这里）**
   - 找到对应功能项，更新状态勾：⬜未做 / 🎨仅前端mock / ✅前后端真接口打通 / ➖epay无我们独有。
   - 这是「某功能精仿到几成」的唯一权威。逐项打勾只在这个文件。

2. **阶段进度 / 下一步 → `docs/开发进度总览.txt`（仅阶段状态变化时才更新）**
   - 只有当一个阶段（如路线图阶段 A~E 的某阶段）推进或完成、或开了新方向时，才回这里更新阶段勾和「下一步」。
   - 顶部「精仿完整度快照」若某大板块整体状态变了（如收单主链从全⬜变基本打通），同步改那几行文字。
   - 总览只打阶段级的勾，不打功能级的勾——功能明细一律看对比清单。

一句话：做完功能先去对比清单打勾；只有阶段性节点才动总览。两边各一套勾，不交叉。

## 收工仪式（每个任务交付后 / 结束工作前必须走完，不可跳过）

> **触发词：用户发「收工」（或「换窗口」「新窗口继续」「结束」）时，立即执行本仪式，无需进一步确认。**

**触发时机**：一个功能/任务做完、用户发「收工」、或**用户说要换新窗口/开新窗口/切窗口继续**时，立即开始走本收工仪式——把当前窗口的活收干净、留痕落盘，再为新窗口准备交接。用户一提「收工」或「换窗口」就等于下达了收工指令，不要等进一步确认。

注意：「收工」触发本仪式（收当前窗口的活）；「开工」触发上面的开工仪式（读规范与记忆、复述、接活）。两个词各管一端，别混。

**必须**按序确认下面每一项，全绿才算真正结束。任一没做 = 未收工，补做后再收：

1. **三关卡确认全绿**：关卡一（epay 完整功能对比，缺失已补齐）、关卡二（构建 vue-tsc / go build 无 error + 设计规范 + 真跑 HTTP 200 无报错 + 停掉临时 server）、关卡三（两轴回写：功能明细进 `docs/epay功能对比清单.txt`，阶段变化才进 `docs/开发进度总览.txt`）。任一没过，当场修复重跑到干净。
2. **回写进度留痕已落盘**：确认对比清单对应功能项状态勾已更新（⬜/🎨/✅/➖），若推进了阶段节点则总览也已同步。没回写视为未完成。
3. **技术债归档**：本次发现但未就地解决的项，已记入 `docs/优化待办.txt`，不留在脑子里。
4. **更新记忆**：本次若产生了新的项目事实/用户偏好/工作反馈（非代码可自证的信息），写进 `MEMORY.md` 对应记忆文件；已失效的记忆及时更正或删除。
5. **清理现场**：删掉验证用的临时文件/临时 server/调试产物，确认 `git status` 无意外残留。
6. **向用户交付小结**：用几句话说清「做了什么、验证到什么程度（构建/测试/真跑结果如实报，失败就说失败）、下一步建议」，不夸大、不隐瞒跳过的步骤。
7. **为新窗口准备交接（换窗口时必做）**：若本次收工是因为要换新窗口，除上面 1~6 外，还要确保下个窗口能无缝接手——把「当前做到哪、下一步该做什么、有无半成品/未提交改动/待办阻塞」写进 `docs/开发进度总览.txt` 的「下一步」区（阶段级）与对应记忆，让新窗口走完开工仪式读到的就是最新交接点。有未完成的半拉子活，明确写清它的状态和续做入口，不留口头交接。

一句话：开工先读规范与记忆并复述；干活走三关卡；收工（含「换窗口」触发）确认三关卡全绿 + 回写 + 归档 + 更新记忆 + 清理 + 如实小结 + 交接留痕。整套闭环每个窗口一律照走。
