## 强制交付门槛（每次写完代码必须走完，全绿才算完成）

一个功能/页面只要动了代码，交付前**必须**按序走完下面两道关卡。任一不通过 = 未完成，当场修复后重跑，循环到干净。不要等用户来指问题。

### 关卡一：与 epay 完整功能对比

写功能时和交付前，都要和 epay 做**完整**功能对比，而不是只看表面：

1. 找到 epay 中对应的源码（`epay.test/admin/*.php`、`epay.test/user/*.php`、插件、模板等），逐项比对。
2. 对齐清单：
   - **字段**：数据字段名、含义、类型是否一致，有无遗漏。
   - **状态字典**：订单/结算/审核等状态的取值与流转是否一致。
   - **操作项**：列表行操作、批量操作、表单动作是否齐全。
   - **业务规则**：金额计算、费率、风控、校验、边界处理是否与 epay 行为一致。
   - **交互流程**：完整走一遍用户路径，确认与 epay 语义一致，mock 数据业务自洽。
3. 发现缺失或偏差 → 补齐/修正后再进入关卡二。

### 关卡二：自检与修复

按序走完，全绿再交付（详见记忆 `self-check-after-feature`）：

1. **构建**：`cd admin-web && npm run build`（vue-tsc 严格 noUnusedLocals，未用 import/type 删干净）。
2. **对照设计规范**：逐条比 UI 设计规范（内容卡片禁 border/ring/shadow/圆角，用浅灰底 `bg-muted/40` 靠色差区分；表格默认左对齐；金额 `¥` 弱化 + `tabular-nums`；用到的颜色类确认已在 style.css 注册；列表 Panel 带主标题 title 不用 flush；开关用 `<Switch>` 组件）。
3. **对照 epay 真实业务**：即关卡一的结果落到代码，字段/状态/操作与 epay 一致。
4. **交互与边界**：分页 clamp、筛选后选中残留清空、菜单关闭、行操作菜单方向自适应、带图标输入框 `!pl-9`。
5. **真跑一遍**：`npm run dev` 起服务，`curl` 目标路由 + .vue/.ts 模块确认 HTTP 200 且编译产物无 error、dev log 无报错。检查完停掉临时 server。

发现问题 → 当场改 → 重新构建验证 → 循环到干净。

### 关卡三：回写进度文档（收尾，不可跳过）

两道关卡全绿后，**必须**按下面两轴分工回写留痕，才算真正完成。不管新窗口还是老窗口，只要完成了功能就必须回写。没回写 = 未完成。

两个进度文档各管一轴，不要混写（这是刻意分工，别把功能明细塞回总览）：

1. **功能完成度 → `docs/epay功能对比清单.txt`（每次做完功能都要更新这里）**
   - 找到对应功能项，更新状态勾：⬜未做 / 🎨仅前端mock / ✅前后端真接口打通 / ➖epay无我们独有。
   - 这是「某功能精仿到几成」的唯一权威。逐项打勾只在这个文件。

2. **阶段进度 / 下一步 → `docs/开发进度总览.txt`（仅阶段状态变化时才更新）**
   - 只有当一个阶段（如路线图阶段 A~E 的某阶段）推进或完成、或开了新方向时，才回这里更新阶段勾和「下一步」。
   - 顶部「精仿完整度快照」若某大板块整体状态变了（如收单主链从全⬜变基本打通），同步改那几行文字。
   - 总览只打阶段级的勾，不打功能级的勾——功能明细一律看对比清单。

一句话：做完功能先去对比清单打勾；只有阶段性节点才动总览。两边各一套勾，不交叉。
